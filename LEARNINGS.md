# Learnings

Lessons captured from past work to inform future development. Updated when merging PRs.

---

## SwiftPM dependencies

- **Neither `branch:` nor `revision:` is a stable pin**: SwiftPM rejects any graph where a version-resolved consumer depends on a branch- *or* revision-pinned transitive dep ("required using a stable-version but ... depends on an unstable-version package"). Only a semver *tag* (`from:`, `.exact`, `.upToNextMinor`) counts as stable. PR #14 pinned `revision: <sha>` believing the SHA was stable, and #12 stayed broken (a SHA is an audit point, not a version). If downstream apps must pin our package by version, every transitive dep needs a tagged release; fork and tag if upstream won't (#13).
- **Forking is the fix when a dormant upstream's latest tag lags master**: `from: "<tag>"` silently regresses past unreleased fixes, and `revision: "<sha>"` keeps the fixes but breaks downstream version resolution (above). The only option giving both is to fork, tag master as a new semver release, and pin the fork by tag (#13). Check `gh api repos/<owner>/<repo>/compare/<latest-tag>...master --jq '{ahead_by, commits}'` to decide whether master is worth keeping over the tag.
- **Verify a resolution fix from the consumer side, not the package side**: `swift build` / `swift test` / inspecting `Package.resolved` from inside the package never apply downstream version constraints. PR #14 passed all three and #12 stayed broken. Reproduce the real constraint: a throwaway downstream package that pins our package via `.exact(...)` and runs `swift package resolve`. Test locally before publishing with local tags and a `file://` URL: clone to a temp path, `git tag v1.x.y <sha>`, then point the consumer at `.package(url: "file:///tmp/clone", exact: "1.x.y")`. Run both a negative control (old tag must fail with the documented error) and positive (new tag resolves) so a green result isn't a false positive.
- **`Package.resolved` is gitignored for libraries in this repo**: Only apps commit `Package.resolved` (they dictate the resolved graph for the whole tree). Don't include it in PR diffs.
- **`swift package resolve` is a no-op if the SHA already matches**: After flipping a manifest from `branch: "master"` to `revision: "<sha>"`, `Package.resolved` may retain the stale `"branch": "master"` hint alongside the revision. To force a clean rewrite: `swift package reset && rm -rf .build .swiftpm Package.resolved && swift package resolve`.
- **Use precise SwiftPM terminology in docs and comments**: API spellings are `.exact("...")`, `.upToNextMajor(from:)`, etc. Xcode's UI labels are "Exact Version", "Up to Next Major". Avoid camel-cased pseudo-names like `exactVersion` or `upToNextMajor`, which imply a SwiftPM API that doesn't exist.

## Evaluating dormant dependencies

- **Audit fork health via the GitHub compare API, not the forks page**: `gh api repos/<owner>/<repo>/forks` lists 20+ forks for any modestly popular repo, almost all stale clones. To find a fork that has actually diverged: `gh api repos/<fork>/<repo>/compare/<upstream>:master...<fork>:master --jq '{ahead_by, behind_by}'`. Anything with `ahead_by: 0` is just a clone.
- **Inventory open issues before forking**: A fork inherits responsibility for upstream's open issues. List them with `gh api 'repos/<owner>/<repo>/issues?state=open'` and the open PRs with `pulls?state=open` so the cost of ownership is explicit before committing.

## Maintaining a forked dependency

- **`gh repo fork` creates the fork with issues disabled**: GitHub's default for forks is `has_issues: false`, so any automation that opens issues on the fork (e.g. an upstream-sync watcher) silently has nowhere to file until `gh repo edit <owner>/<fork> --enable-issues`. Verify with `gh api repos/<owner>/<fork> --jq '.has_issues'`.
- **`gh repo clone <fork>` auto-adds an `upstream` remote** pointing at the parent (plain `git clone` does not), so a follow-up `git remote add upstream ...` errors with "remote upstream already exists". Use `git remote get-url upstream >/dev/null 2>&1 || git remote add upstream <url>` to be idempotent.
- **A maintenance commit on master ships inside the version tag**: tagging the fork's master HEAD bundles any infra commits (CI workflow, README) into the tag. Harmless for SwiftPM, which only consumes `Sources/`, but "content-identical to upstream" is then true only of the library sources, not the repo. Write "library sources identical" in changelogs to stay accurate.
- **An upstream watcher must fail loud, not silent**: a scheduled job that detects upstream movement should `git rev-parse --verify` both refs and assert the commit count is numeric before acting, else a wrong ref range reports "0 behind" and the run goes green while upstream has actually moved. GitHub Actions runs `run:` blocks under `bash -eo pipefail`, which catches hard command failures but not a wrong-but-successful `0`. Narrow `|| true` and `2>/dev/null` to the one expected condition so real failures still surface.

## Cross-platform Foundation

- **`CFString*` IANA-registry APIs are Apple-only**: `CFStringConvertIANACharSetNameToEncoding`, `CFStringConvertEncodingToNSStringEncoding`, `CFString`, `kCFStringEncodingInvalidId` are not in swift-corelibs-foundation. Anything that imports them needs a `#if canImport(Darwin)` guard with a sensible non-Apple fallback (UTF-8 is usually fine for charset-resolution paths). Linux compile failures here can sit hidden if the only Linux CI job runs against external services that fail for unrelated reasons (see `CI workflow design` below).
- **`String.Encoding` coverage on Linux is uneven**: Named constants like `.shiftJIS`, `.iso2022JP`, `.japaneseEUC`, `.utf16LittleEndian`, `.utf16BigEndian` *compile* on Linux (constants exist in swift-corelibs-foundation), but `String(data:encoding:)` only successfully decodes some of them; coverage depends on the platform's ICU build. In `swift:5.10` Docker, Shift-JIS / ISO-2022-JP / UTF-16LE/BE work; EUC-JP returns nil. Don't assume "compiles on Linux" means "works on Linux" for these encodings.
- **For tests that exercise platform-divergent behaviour, prefer conditional asserts over skipping**: `#if canImport(Darwin) ... #else ... #endif` inside a single test documents what the platform actually does (e.g. EUC-JP decodes to `あ` on Apple, passes through verbatim on Linux). Skipping hides the contract; conditional asserts pin both behaviours so a regression in either platform fails CI.

## CI workflow design

- **A separate `swift build` / unit-test job for each platform is non-negotiable**: A single integration job that depends on Docker services (e.g. GreenMail) can hide compile failures behind unrelated service problems. PR #18 / v1.2.2 / v1.2.3 shipped a Linux compile bug because `greenmail-integration` was the only Linux job and its red status looked like flake. The fix (PR #20): add a plain `test-linux` job (no services) that just builds and runs the unit suite. Apply the same pattern for any new platform-target.
- **A red CI run can still be merged**: branch protection has to explicitly require each check by name. Adding a job to `release.needs:` in the workflow only gates the *release* step, not the merge button. PR #18 merged red because `greenmail-integration` wasn't a required check at the repo level. Worth verifying required-check settings periodically.
- **PR review for substantive changes**: `code-reviewer` agent on the diff before opening the PR catches issues that would otherwise come back via Copilot. Cheap insurance.
- **A long-lived release branch needs its own PR trigger**: `pull_request: branches: [main]` meant every PR into `version/v2.0` ran zero CI — the entire #27/#28 line merged untested, and nobody noticed because "no checks reported" reads as "pending" on the PR page rather than "never triggered". When creating a `version/*`-style branch, grep `.github/workflows/*.yml` for `branches:` filters first; the fix is `branches: [ main, 'version/**' ]` on `pull_request` (leave `push` alone to avoid duplicate runs).
- **Close/reopen does not reliably re-trigger workflows**: reopening five PRs created runs for only three, and a second close/reopen of the failing two produced nothing again, with no discernible state difference (PRs OPEN, MERGEABLE/CLEAN). An empty commit (`git commit --allow-empty`) fires a `synchronize` event, which triggers every time. Don't burn time diagnosing the reopen path.
- **A `CANCELLED` check is usually runner eviction, not failure**: `test-macos` cancelled mid-step with no concurrency config in the workflow and the same suite green on Linux/GreenMail. `gh run rerun <id> --failed` re-runs just the cancelled job.

## `gh` CLI tricks

- **Per-job logs are available while the run is still in progress**: `gh run view --log-failed` waits for the whole workflow to complete. `gh api repos/<owner>/<repo>/actions/jobs/<job-id>/logs` returns immediately for any completed job, even while sibling jobs are still running. Find the job id via `gh api repos/<owner>/<repo>/actions/runs/<run-id>/jobs --jq '.jobs[] | select(.name=="<job>") | .id'`.
- **`gh run rerun <id> --failed`** re-runs only the failed jobs of a completed run, not the whole workflow. Won't work while the run is still `in_progress` (you'll get "cannot be rerun; This workflow is already running"), so wait for the run to finish first.
- **`gh release edit --notes-file <path>`** retrofits release notes on an existing tag without retagging. Useful for adding ⚠️ notes to releases that shipped with platform-specific bugs (we did this for v1.2.2 and v1.2.3 rather than yanking). Tags, CHANGELOG.md, and downstream consumers stay untouched.
- **Pushing a tag auto-creates the GitHub Release in this repo**: the `release` job in `.github/workflows/ci.yml` runs `softprops/action-gh-release@v1` with `generate_release_notes: true` whenever a `refs/tags/*` ref is pushed. The AGENTS.md recipe says `gh release create v1.x.x --notes-file ...`, which errors with `HTTP 422: tag_name already exists` because the workflow won the race. Use `gh release edit v1.x.x --notes "$(...)"` afterward to overwrite the generic auto-notes with the CHANGELOG entry.
- **`gh issue develop <n> --checkout` makes ugly branch names**: e.g. `21-deterministic-linux-timeout-in-testfetchmessagebodyfindscorrect...` (the full issue title kebab-cased). Rename immediately: `git branch -m <better-name>`, then `git push origin -u <better-name>` and `git push origin --delete <ugly-name>`.
- **`gh run view --log-failed` truncates around the failure annotation**: it returns context near the failure marker, which can omit the actual test output if tests run from line ~1650 but the annotation lives at line ~1457. Full per-job logs: `gh api repos/<owner>/<repo>/actions/runs/<run-id>/logs > logs.zip && unzip logs.zip`; each job lands as a numbered `.txt` with everything captured.
- **`gh pr checks` exit codes**: 0 = all passing, 1 = at least one failed, 8 = at least one still pending. Use this in scripts instead of grepping the text output. `gh pr checks <n> --watch --interval 30` blocks until all checks finish, which suits a background "merge when green" task.
- **`mergeStateStatus` is computed asynchronously and is stale right after a force-push**: immediately after a rebase + force-push, `gh pr view --json mergeable,mergeStateStatus` can report `CONFLICTING`/`DIRTY` even when there's no real conflict. Verify locally instead: `git merge-base --is-ancestor origin/main <branch>` returning success proves `main` is fully contained and the merge will be clean. Re-query the API a few seconds later and it settles to `MERGEABLE`/`UNSTABLE`.
- **Fast-forwarding a PR's commits into its base auto-marks the PR as MERGED**: when `version/v2.0` was FF-merged to include the `#32` branch's head, GitHub flagged PR #32 as `MERGED` (state, not just closed). You then *cannot* `gh pr close` it ("already merged"). Useful when folding one PR into another before rebasing the base.
- **Re-request a Copilot review**: `gh pr edit <n> --add-reviewer copilot-pull-request-reviewer`. The pending request only shows via GraphQL as a `Bot` reviewer (`reviewRequests.nodes[].requestedReviewer ... on Bot { login }`); the REST `reviewRequests` JSON field comes back empty for it.

## Test infrastructure

- **`MockIMAPServer` mutable state is serialised behind a `NIOLock`** (#21, PR #24): every property touched by both the test task (`setResponse`, `setAuthenticate*`, reads of `receivedCommands`/`receivedContinuations`) and the NIO event loop (`channelActive` → `greeting()`, `handleCommand`) lives under `lock`. `handleCommand` acquires once for the whole call: only string parsing inside, no I/O, so the held duration is fine. `channel` is intentionally outside the lock (test-task-only). Don't add a new mutable property without deciding which side of that boundary it sits on.
- **A 5.005-second test runtime is a smoking gun**: it matches the `waitForGreeting` timeout at `Sources/SwiftIMAP/Networking/ConnectionActor.swift:456`. If a test fails with `connectionFailed("Operation timed out")` and runs for ~5s, the TCP connect almost certainly succeeded: the IMAP greeting just never arrived. Look at the mock server's `channelActive`, not the network layer.
- **"Deterministic on platform X, fine on platform Y" is not evidence of platform-specificity**: #21 was originally framed as a Linux-only race because it failed 100% on Linux Docker and passed in 7 ms on macOS. The same race later bit macOS too (PR #20's CI hit `testAuthenticateOAuth2UsesSaslMechanism` on Linux and `testFetchMessageReturnsNilWhenUIDMismatch` on macOS in the same run). A race that fires 100% on one platform and ~1% on another isn't platform-specific: it's one race with different scheduling. Don't reach for `#if os(...)` guards as a fix; fix the race.
- **`MockIMAPServer` lives in its own file** (`Tests/SwiftIMAPTests/MockIMAPServer.swift`, extracted in #28): it was inline at the bottom of `IMAPIntegrationTests.swift`, and adding ~25 lines of mock plumbing tipped that file past the `file_length` lint limit (error at 1200). `file_length` is an *error*, not a warning, so it fails the build/lint gate. Keep shared test infra in its own file; don't grow `IMAPIntegrationTests.swift`.
- **To test connection-teardown paths, give the mock a "close after response" hook**: `MockIMAPServer.closeAfterResponse(toCommandContaining:)` sends the configured (untagged) line then closes the channel, so a client command stays pending until the hang-up. This is how the mid-session `* BYE` surfacing (#26 follow-up) and the greeting→persistent-handler window get a deterministic test. Note the mock normally auto-appends a tagged `OK` to any `*`-prefixed response; suppress that for close-after commands or the command completes instead of hanging.
- **An "end-to-end" acceptance criterion is load-bearing — classification tests are not a substitute**: #35's criterion said an abrupt drop on a *wrapped operation* must reconnect and retry. The classification-level tests all passed while the feature had never worked: a second, independent defect (connection state never reset on channel death) only surfaced when the end-to-end test was actually written. If the spec says end-to-end, write end-to-end first and let it drive out the hidden defects.
- **Reconnect tests need a one-shot hang-up**: `closeAfterResponse` re-fires on the retried command, so a reconnecting client can never succeed. `closeOnceAfterResponse(toCommandContaining:)` consumes itself after the first match, letting the second attempt complete — the difference between testing "fails on drop" and "recovers from drop".
- **Count-based CI assertions can't prove semantic caching — serve different data**: asserting "exactly 2 CAPABILITY commands" doesn't show the post-auth set is what's cached. `MockIMAPServer.setResponseSequence(for:responses:)` serves a reduced pre-auth set and a full post-auth set; the test then proves MOVE gating used the second (UID MOVE chosen, no COPY fallback).

- **`gh issue develop --base <branch>` does not reliably branch from `<branch>`**: with `--base version/v2.0` the issue branches were created at that branch's tip, but with `--base <feature-branch>` (stacking) the branch was created from `version/v2.0` instead. Always `git log --oneline -1` after `--checkout` and `git reset --hard <intended-base>` if the start point is wrong (cheap while the branch has no commits).

## Reviewing Copilot suggestions

- **Always evaluate against actual evidence**: Copilot suggested removing `--skip GreenMailIntegrationTests` because "swift test option support varies across SwiftPM versions", but we'd just observed `--skip` succeed twice on `swift:5.10` in this exact CI job. The test-coverage suggestion was a real gap (5 new charset cases with no tests); the `--skip` suggestion had a flawed premise. Reply on PR with the rationale for any skipped suggestion so the next reader sees why.
- **COW-shared arrays leak the lock**: `lock.withLock { _array }` returns a Swift array that still shares copy-on-write backing with `_array`. The caller iterates outside the lock; a later append inside the lock triggers a COW copy and (per Swift's docs) concurrent access to non-`Sendable` value types is undefined behaviour even with COW. Force a fresh snapshot inside the lock: `Array(_array)` or `_array.map { $0 }`. Caught by Copilot on PR #24.
- **The "won't compile / won't match" optional-pattern claim is a false positive**: Copilot (PR #28) flagged `if case .tryCreate = code` and `if case .other(let n, _) = response.code` (where `code` is `ResponseCode?`) as broken. Matching a non-optional enum-case pattern against an `Optional` is valid Swift: it matches only `.some(pattern)`, never `.none`. The build was green and tests proved the match worked. Don't rewrite to `.some(...)`/`...?` on the bot's say-so; verify against the build + a test that exercises the branch, then resolve as won't-fix with the rationale.
- **Copilot reviews a specific commit and goes stale after a force-push or new pushes**: its review carries a `commit.oid` (`reviews.nodes[].commit.oid` via GraphQL). After a rebase/force-push it does not auto-re-review. Compare that oid to the PR head to detect staleness; re-request with `gh pr edit <n> --add-reviewer copilot-pull-request-reviewer`, then poll until a review whose `commit.oid` equals the head appears.
- **Settle concurrency-semantics claims with a ten-line probe, not an argument**: Copilot claimed a cancelled awaiter of `Task.value` could fire a `defer` early and break connect-coalescing (PR #45). A 30-second scratch script (cancel the awaiter at 0.1 s, watch it still receive the value at 1.0 s) refuted it conclusively. Reply with the probe output, document the load-bearing property at the code site, and resolve.
- **Trace the claimed call order in source before applying a reviewer's fix**: an internal review agent flagged the PREAUTH path as leaving a stale capability cache (#36). Reading `connect()` showed the `CAPABILITY` command issued after a PREAUTH greeting already runs in authenticated state, so the cache was correct by construction. Reject with evidence and pin the question with a test so it doesn't recur — same discipline as Copilot false positives.
- **When a reviewer flags a missing guard, also audit its position against every side effect**: the UIDPLUS guard existed but sat after the `STORE` (stranding `\Deleted` flags) and after `SELECT` (mutating selected-mailbox state on a doomed call). The fix is ordering, and the tests must assert the *absence of each prior side effect* (no STORE, no SELECT reached the server), not just the final outcome.

## Error and API design (security)

- **Never interpolate a command/error enum that carries secrets**: `IMAPError.commandFailed` built its message with `String(describing: command)`, so a rejected `LOGIN` embedded the cleartext password (and `APPEND` the whole message body) into the error *and* its `localizedDescription` (#27). The same `String(describing:)` leak hit the outgoing-command debug log and the `invalidState` validator messages. Fix: an argument-free `IMAPCommand.Command.label` (the bare verb, e.g. `"UID MOVE"`) used everywhere a command appears in human-facing text. When adding any error/log line that touches a command or request, use the label, never the value.
- **Synthesized reflection leaks associated values in logs too, not just errors**: `logger.log(.trace, "Handling response: \(response)")` dumps full `FetchAttribute` payloads (message bodies, headers) at trace level because `IMAPResponse` has no `CustomStringConvertible`. Audit every `\(someEnum)` interpolation that can carry `Data`/credentials, in log statements as well as errors. (Tracked for follow-up in #29.)
- **Prefer one structured error + accessors over many bespoke cases**: `IMAPError` had `mailboxNotFound`/`messageNotFound`/`quotaExceeded`/`permissionDenied` cases with *zero* producers (confirmed by grepping construction sites). They fragmented consumer handling for failures that all arrive as a server `NO`. Replaced with a single `commandFailed(IMAPServerResponse)` plus semantic accessors (`isMailboxNotFound`, `isOverQuota`, ...) keyed off the RFC 5530 response code. Grep for producers before keeping an aspirational enum case.
- **Truncate untrusted server input before putting it in an error**: parser errors that interpolate the offending line can dump unbounded content (incl. message data) into logs. `String.truncatedForDiagnostics(limit:)` caps it while preserving the total length. Only ~6 of the 50+ `parsingError(...)` sites actually interpolate input; the rest are static strings, so fix the interpolating ones rather than blanket-wrapping.

- **Audit every error case for producers — dead cases accumulate**: across two passes, 7 of 17 `IMAPError` cases had no `throw` site anywhere (#27 removed four, #35 three more). Consumers were exhaustively handling errors that could never occur. `grep` per case across the source is a ten-minute check that belongs in any API review.
- **Two classifiers over one error space drift apart**: `isRetryableError` and `requiresReconnection` partitioned `IMAPError` independently; abrupt drops ended up neither retryable nor reconnectable, and definitive BYEs were reconnected (discarding the server's reason) when retry classification said never retry them. When touching one classifier, diff its case list against the other.
- **Wrap foreign errors into the typed error at the transport boundary**: `errorCaught` dispatched raw NIO/SSL errors to pending commands, which silently bypassed every `as? IMAPError` classification downstream (no retry, no reconnect). Any boundary where third-party errors enter must convert to the library's typed error, preserving the original as `underlying`.

## Swift and NIO concurrency

- **A handler invoked while a `NIOLock` is held must not call back into the locked API**: `NIOLock` is non-reentrant. The obvious "clear the one-shot handler by calling `setResponseHandler(nil)` from inside the handler" deadlocks, because the handler runs under the same `lock.withLock` that `setResponseHandler` takes. The one-shot fix (#26) instead clears `_responseHandler` *directly* under the already-held lock (a `oneShot` flag checked in `dispatch`/drain), never via the public method. When a callback needs to mutate the lock-protected state it runs under, mutate the ivar directly, don't re-enter the accessor.
- **Adding an associated value to an enum case is source-compatible for matchers**: changing `case connectionClosed` to `case connectionClosed(IMAPServerResponse?)` does *not* break `case .connectionClosed:` in switches or `if case .connectionClosed = e`, because pattern matches can ignore associated values. Only *construction* sites (`.connectionClosed` as a value) need `(nil)`. Lets you enrich an error case while touching only the few places that build it.

- **`Task.value` is not responsive to the awaiting task's cancellation**: a cancelled awaiter still suspends until the underlying task completes and receives its result (verified empirically, Swift 5.10). This makes "store the in-flight `Task`, late callers await its `.value`" a sound coalescing pattern on an actor — the stored task cannot be cleared early by a cancelled waiter. Document the property at the `defer` that relies on it.

## IMAP protocol

- **A `* BYE` before a LOGOUT's tagged `OK` is normal, not an error**: surfacing an unsolicited mid-session `BYE`'s reason must be deferred to connection teardown (`channelInactive`), not applied when the `BYE` arrives, otherwise it wrongly fails a pending `LOGOUT` (which legitimately receives `* BYE` then `OK`). Capture the BYE code/text in a `pendingBye`, and only convert pending commands to `connectionClosed(IMAPServerResponse(.bye, ...))` once the socket actually drops, by which point a normal LOGOUT has already completed (#26 follow-up, #28).

- **Capability tokens are case-insensitive (RFC 3501 §7.2.1)**: `Set.contains("MOVE")` against tokens stored as received silently selects the COPY fallback (or throws `unsupportedCapability`) for a server advertising `move`. Normalise to upper case once at the cache-write boundary so every gate (MOVE, UIDPLUS, LITERAL+, SASL-IR, LOGINDISABLED) compares against upper-case literals.
- **The capability set can differ pre- and post-authentication**: servers commonly advertise a reduced set before login and don't always re-announce in the LOGIN/AUTHENTICATE completion. If operations gate on a *cached* capability set, refresh it once post-auth in `connect()`. PREAUTH needs no extra refresh: a CAPABILITY command issued after a PREAUTH greeting already runs authenticated.
- **Never fall back from `UID EXPUNGE` to bare `EXPUNGE`**: the bare form expunges *every* `\Deleted` message in the mailbox, not the requested UIDs — silent data loss that looks like a graceful degradation. Without UIDPLUS, the only safe targeted-expunge behaviour is to throw.

## Git: rebasing long-lived branches

- **Rebasing a long-lived branch onto `main` conflicts on `CHANGELOG.md` at every commit that touched `[Unreleased]`**: each replayed commit re-applies its `[Unreleased]` edit against a section other PRs have since changed, so you resolve the same region 2+ times during one `git rebase`. Resolve by combining entries each time. Because this repo squash-merges PRs, a `git merge main` into the branch would resolve it once and the final squash flattens the merge commit anyway, so rebase only buys a linear branch history that the squash discards. Choose merge unless you specifically want the linear branch.
- **`GIT_EDITOR=true git rebase --continue`** completes a conflict-resolution step without dropping into an interactive editor (this harness can't drive `git rebase -i` or editor prompts). Same trick for any git command that would otherwise open `$EDITOR`.

## API review method

- **Audit the real consumer before classifying breaking changes**: the v2.0 review read MailTriage's source and inventoried every SwiftIMAP symbol it touches. "Limit breaking changes" became a *measured* cost (two files it already had to touch) instead of guesswork, and the unused ~80% of the public surface became freely reshapeable. The inventory also surfaced the workaround list (client rebuilds, STATUS-before-write, bracket-stripping) that drove the highest-value fixes.
- **Run independent spec-check and code-review agents per PR, and expect both verdict types**: across five PRs the spec-checkers caught a load-bearing test gap (the dead reconnect path) and the code-reviewers caught real defects (guard ordering, memory doubling, case-sensitivity) *and* produced false positives (PREAUTH cache, Task.value cancellation). The findings are leads, not verdicts — every one was verified against source or a probe before acting.

## Multi-agent review sessions

- **Background agents must diff refs, never check out branches**: an agent running bare `git checkout <branch>` can detach or steal the main worktree's branch, and the session's shell cwd can silently end up inside a leftover `.claude/worktrees/agent-*` directory (relative-path commands then operate on the wrong tree while absolute-path tools hit the main one — apparently contradictory file states). Prompt agents to use `git diff ref...ref` or `git checkout --detach`; after agents complete, verify `pwd`/`git rev-parse --show-toplevel` before any git work, and `git worktree remove --force` the leftovers. Worktrees share the object store, so commits made from the wrong worktree still land on the right branch — the damage is confusion, not loss.
